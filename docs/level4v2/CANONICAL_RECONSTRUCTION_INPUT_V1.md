# Level 4 v2 Canonical reconstruction input V1

## Purpose

Rebuild Level 4 from the capabilities actually proven by the Spiral program. The reconstruction must remove experimental duplication while preserving authority, determinism, Recovery and observation boundaries.

## Canonical facts to carry forward

### Meaning and invocation

- Semantic meaning is separate from GPU execution representation.
- Runtime-varying values are explicit verified inputs, not Backend guesses.
- Active membership, modified membership and final game operations remain external responsibilities.
- Indirect quantity, temporal history and sparse membership can coexist under one exact invocation authority.
- `A_t`, `M_t`, `N_t`, `R_t`, `W_t`, `H_t` and `T_t` are exact sets with versioned canonical serialization.

### Candidate authority

```text
Canonical Semantic
  -> Raw Candidate
  -> Planner
  -> Planner-independent Verifier
  -> opaque Verified Plan
  -> Frozen Artifact
  -> Runtime materialization
```

Raw Candidates cannot execute, freeze themselves, mint a seal or select policy. The Verifier independently reconstructs expected operations, resource identities, write sets and Target bindings.

### Composition

- Composition owns producer/consumer relations, flow identity and schedule authority.
- Runtime and Backend do not reinterpret membership, action, Candidate or history policy.
- The Level 4 v1 proven subset remains Buffer-only, finite static DAG, single writer, zero-or-one presenter, one Adapter and whole-composition Recovery.
- v2 must not silently claim Texture sharing, cycles, conditionals, variants, streaming, residency or multi-device support.

### Dynamic execution

The proven dynamic dimensions are distinct strong concepts rather than interchangeable integers:

```text
Active count
Transition count
Reuse count
Update interval
Batch size
Affected block count
Timeline ordinal
Device epoch
```

The canonical model must retain their semantic roles and prevent accidental substitution.

### History and Recovery

- History validity is explicit and bound to Semantic generation and Device epoch.
- Recovery invalidates representations, completion state, readback state and history handles.
- External current state must be rebound explicitly.
- The first post-Recovery materialization performs a verified full-active seed with no stale retention.
- Removed Adapter identity is excluded from retry.

### Measurement

- WARP proves correctness; real hardware supplies observational performance evidence.
- Measurement profiles, Adapter fingerprint, Shader digests and Candidate order are identity-bound.
- Median ordering is descriptive when paired evidence is the authority.
- Noise-equivalent and unresolved are first-class outcomes.
- Performance evidence cannot directly authorize Runtime selection.

## What must not enter the Canonical core

- Spiral numbers and CU numbers in public runtime types.
- RTX 4070-specific thresholds or winner rules.
- Fixed 4096/128/64 values unless they belong to a versioned qualification profile rather than the generic type.
- HLSL source, filesystem paths, timestamps, random UUIDs or process-specific data in Frozen execution identity.
- Game-specific final-operation policy.
- Hardware-specific policy disguised as Semantic meaning.
- Parallel copies of the same fact across Semantic, Candidate, Plan and Runtime records.

## Reconstruction success condition

A v2 implementation is not accepted because it compiles or looks cleaner. It is accepted only when a migration corpus reproduces the relevant accepted identities, observations or deliberately versioned replacements, and when every changed invariant has an explicit supersession record.
